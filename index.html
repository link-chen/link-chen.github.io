引用和指针的区别

引用是一个变量的别名不是实体故不能独立存在，在使用引用时必须初始化，而指针不需要初始化

example

int a=0; int &b=a; (√)

int &a;(×)

自增运算时不同

example

int a=0,*b=&a;

b++;实质上是内存指向的区域往后便宜了一个int的内存大小

int a=0,&b=a;

b++;此时b=1;

指针可以为空而引用不可以

malloc和free搭配

凡是new就必须delete，否则容易内存泄漏（当然可以考虑智能指针）

运行时多态如何实现

1.要求基类对应的方法必须声明为virtual方法

2.此时基类和父类的实际内存大小会比按照变量的计算多8Byte（64位机）->多出一个虚指针

3.在编译时会生成虚函数表（存放在常量区）

4.调用对应方法时依靠虚指针到虚函数表中查找真正应该执行的方法

Cast<Target*>(Src)

​ 只要没有底层const，就可以用此转换

​ 可以关闭一些精度损失警告，转换（void *）

​ 专用于转换有底层const的变量

3.reinterpret_cast<>()

​ 很危险的操作，用于提供更加底层的转化

​ interpret是诠释的意思，那顾名思义reinterpret就是重新诠释这个区域

​ 依据此方法进行的类型转换，会在指针的底层解引用成Target类型的

​ 例如大端存储的int a=0x61620000 本质上是整数，但如果通过reinterpret转换，打印的a会是“ab”(详情参见ASCII码)

4.dynamic_cast<>()

​ 用于进行运行时多态的一些转换

​ 常见的输入有：

​ dynamic_cast<&&>(e); //&&用于右值

构造函数与析构函数

1.构造函数不能设置为virtual，此时虚函数表尚未填入对应的方法，故不能实现多态

2.析构函数默认为是可以重写的，且一般建议设置为virtual（防止部分编译器没默认允许析构函数重写）

3.除构造函数以外的一切非static函数，均可以设置为virtual

拷贝构造函数和移动构造函数

首先明确哪些情况下会使用拷贝构造函数？

1.有等号发生时（需要重载赋值运算符支持这一行为）

2.函数传入的是一个类，但既不是指针也不是引用时

3.返回一个类的对象，同样不是指针或者引用时

2.移动构造函数是来骗，来偷袭，直接抢走t的部分内存区域的指针，并将t的指针指向区域为null

What is Rendering PipleLine?

一个输入为3D场景，输出为二维图像的工作流

渲染管线的阶段

​ 1.应用阶段：通常由CPU实现。准备场景数据（摄像机位置，视锥体等），粗粒度的剔除（为提高渲染性能），设置模型的渲染状态，输出渲染所需的几何信息（最重要的任务）

​ 2.几何阶段：逐顶点，逐多边形操作

​ 3.光栅化阶段：最终生成图像，决定每个渲染图元的像素是否能被绘制（绘制在屏幕的位置）

CPU执行DrawCall指令后，渲染就从应用阶段进入后续阶段，应用阶段CPU主导，此后为GPU主导

GPU流水线

几何阶段

顶点着色器：进行顶点的坐标变换，计算顶点的颜色，将顶点坐标从模型空间转到齐次裁剪，很重要，一般都要写

曲面细分着色器：很少用

几何着色器：很少用

裁剪：去除一定不会最后显示的部分

屏幕映射：将点片元映射到屏幕上

光栅化阶段

三角形设置：依据几何阶段输出的顶点，生成三角形

三角形遍历：依据上一阶段生成的三角形判断三角形覆盖了哪些像素

片元着色器：fragment

逐片元操作：进行模板测试（不通过就废弃），深度测试（Z-Buffer），混合等操作，最后输出到颜色缓冲区

Phone And Blinn-Phone

均将着色区域当作一个平面（因为区域都很小，应用微分的思想）

均将环境光当作常数

计算漫反射时也相同，区别仅在处理高光项时有所不同

Blin-Phone 光照模型使用了半程向量进行计算高光项

Phone 光照模型则是直接通过计算视线和反射光线的夹角计算高光

定义如下

着色平面法相$\vec{n}$，入射光线$\vec{l}$，反射光线$\vec{r}$，观察方向$\vec{v}$，入射光线与法线的夹角 $\theta$，环境光强度Ia

表面对于各个光的反射率分别为Ka,Kd,Ks(环境光，漫反射，高光)

Blinn-Phone:

$\vec{h}$=normalize($\vec{l}+\vec{r}$) $\vec{h}$为半程向量，提高计算效率

Is=Ka*Ia+Kd*max(0,$\vec{l}\vec{n}$)+Ks*max(0,$\vec{n}\vec{h}$)^$\alpha$

Phone:Ka*Ia+Kd*max(0,$\vec{l}\vec{n}$)+Ks*max(0,$\vec{r}\vec{v}$)^$\alpha$

we use Ks to multiply pow(max(0,$\vec{n}\vec{h}$),cos$\alpha$),so why we use pow?

使用乘积的$\alpha$次方,用以快速衰减高光项范围$\alpha$越大高光范围越小

一般而言，对于写实类的，Phone光照模型更真实，但Blinn-Phone效率更高（反射光和观察方向的夹角不好算）

快速排序

快速排序原理：选取一个pivote，作为标杆，将小于该数的数放在pivote前，大于该数的放在pivote后面

不断划分两个区间，左侧为小于该数的，右侧为大于该数的

直到左右两个区间不再存在时，此时数组已经有序

时间复杂度O(nlogn)，是不稳定排序

时间复杂度O(nlogn)，空间复杂度O(Ologn)

Computer Internet

自下而上的网络体系结构

OSI 7层体系结构：物理层，数据链路层，网络层，传输层，会话层，表示层，应用层
TCP/IP四层模型：网际接口层，网络层，传输层，应用层
TCP/IP 网际接口层=OSI(物理层+数据链路层)
TCP/IP 网络层=OSI(网络层)
TCP/IP 传输层=OSI(传输层)
TCP/IP 应用层=OSI(会话层+表示层+应用层)
OSI为法定的规则，但TCP/IP是事实上的规则

OSI 各层的功能+设备+协议

物理层：

确定物理设备的特性（电气特性，机械特性等），传输的是比特
物理层设备：集线器，中继器。集线器的各个端口会平分设备总带宽（复用）
物理层信道：单工，半双工，全双工

数据链路层：

将数据封装成帧，标识开头和结尾，透明的传输数据帧，差错检测（奇偶校验，CRC）
MTU是值得最大传输单元
数据链路层有两个子层，自下而上是MAC子层和LLC子层
MAC是设备的唯一地址局域网内可以通过ARP协议获取MAC地址
MAC子层将比特流封装成帧
LLC子层实现数据链路层的硬件无关性
设备：网桥，交换机（交换机可以划分冲突域）

接入控制防止冲突的方法

1.ALOHA 直接发送

2.CSMA 信道为空时发送，否则放弃发送

3.CSMA/CD 载波侦听多点接入碰撞检测（信道为空就发送，否则等一段时间再来，直到尝试到一定次数后放弃）

4.CSMA/CA ……碰撞避免

网络层：

实现两个端到端的数据传送，寻址+路由选择
设备：路由器（一个路由器分隔一个广播域，网络层以上的设备都是网关）
协议（IP,ICMP,ARP）
IP:IP地址头20字节，其中4字节为版本号等，标志位MF=1时后续有分片，MF=0后续无分片。DF=0时允许分片，DF=1时不允许分片
ICMP:ping会调用ICMP协议，会告知目标端的一些错误（原点不可达，路由改变等）
ARP:通过广播，发送包含目标设备的IP地址的数据包，请求获取其MAC地址。当对应设备收到该数据包后，便会向ARP请求方返回一个包含MAC地址的数据包

传输层：

​ 补充一个socket={IP,端口号}
​ 设备：网关（传输层及以上均为网关）
​ 协议：TCP，UDP
​ UDP：头部8字节（16源端口号+16位目的端口号+16位UDP长度+16位校验和）,提供无连接，不可靠的数据传输服务。速度快，但不可靠，不能保证顺序，不能保证数据无丢失，可以用于广播

​ ##### TCP三次握手和四次挥手（此处应该有图片）

​ 挥手:客户端发送（FIN=1,seq=x）,服务器收到后回复（Ack=x+1,seq=y）,服务器向客户端发送（FIN=1，Ack=x，seq=y）,客户端发送（Ack=y+1，seq=z）
​ TCP传输中，接收到一个末尾为seq=x时，反馈的Ack=x+1

TCP拥塞控制

1.什么是拥塞控制

​ 在网络中发生拥塞时，减少向网络中发送数据的速度，防止造成恶性循环，在网络空闲时提高发送数据的速度提高发送数据的速率，最大限度的利用网络资源

2.TCP的拥塞控制方法

​ 1）TCP滑动窗口

​ TCP不同于UDP发送的是一个个数据段，采用流水线的方式（渲染管线有端联想）。TCP维护一个发送窗口来发送数据，这个窗口就是一个区间。在窗口内的数据会被一次性发送，不需要等待之前发送的数据段被确认，当最早发送出的数据段被确认，窗口就会向前移动。未收到按顺序确认的序列号的话，窗口仅会移动到有序确认的最后一个序列号的位置。

​ 2）慢启动

​ 3）拥塞避免

​ 4）快速恢复
